bayeswire / docs / toolchain

A Bayesian workflow an agent can run end-to-end through files

Deterministic, replayable, and auditable. The customer is an agent — and the human auditing it — so the product is trustworthy process, not a sampler. This page explains the philosophy, walks a model from Python class to plots through jaxstanv5 and bayesite, and shows how CI holds the contract together.

philosophy

Three commitments

The contract is the product

The system's real interface is not any one tool — it is the model language and the artifact formats the tools exchange. bayeswire is that language: spec, corpus, and conformance tests versioned together in one place.

One normative copy of the truth. Every other repository is a producer or consumer that provably conforms; drift is a red CI job, not an archaeology finding.

The model is data, not code

A run directory that bottoms out at “a hash of arbitrary Python” needs untrusted code re-executed to be interpreted. Instead, models serialize to an IR document, and meta_from_dict decodes it while executing no user code.

Every downstream phase — diagnose, posterior-predictive, a re-fit months later — proceeds from the artifact alone. The IR is an audit-trail primitive first, an interchange format second.

Minimal trust surface, honestly stated

The default path executes your model in an environment containing exactly one stdlib-only package plus one auditable Rust binary. Not “no supply-chain danger” — rather: every step is small enough to audit, pinned enough to reproduce, deterministic enough to replay.

Where backends differ in capability, the difference is recorded and tested, never hidden.

the repositories

One language, two engines, one harness

Dependency direction is one-way and explicit. Solid arrows are pinned package dependencies (exact commit — the pin-bump diff is the compatibility review). The dashed arrow is file vendoring: bayesite takes bytes, never packages.

Nothing depends on a backend to define a model. bayeswire imports nothing outside the standard library — a JAX import anywhere in it is a bug by definition, enforced by a test that walks every module.

the contract

The wire format at a glance

A model declaration resolves to ModelMeta — the serialization boundary — and serializes as one versioned JSON document:

{"bayeswire_ir": 1, "model": { "node": "ModelMeta", … }}

The rules that make it trustworthy, in one breath: node tags and field lists are the contract, never Python class names; entry-array order is semantic and must never be reordered (free_values is the flat NUTS state layout); decoding executes no user code; unknown non-core tags fail explicitly. The canonical serialization is byte-exact, and the model hash is sha256(canonical_bytes(meta)) — computed by the producer at write time. Consumers hash the file as received and never reserialize to hash, which sidesteps cross-language float-formatting divergence between Python and Rust.

Dimension labels (Dim, coordinates) never enter the IR: they travel in a separate sidecar document, so two models differing only in labels share one model hash. The normative prose lives in spec/ir-format-v1.md; downstream copies are generated, hash-checked bytes.

the integrity seal

model_data_fingerprint: one hash, every tool

A run directory is only as trustworthy as the claim that its fit came from these exact files. One hash binds a fit to the exact bytes of the model and data documents it was sampled from — and every tool in the toolchain, Python or Rust, computes it the same way.

The model document is the canonical IR bytes; the data document is the canonical bayescycle.data.json.v1 wrapper (spec/data-document-v1.md). The definition lives in spec/model-data-fingerprint-v1.md, with bayeswire.ir.model_data_fingerprint as the reference implementation.

the one rule

Producers hash the bytes they write. Verifiers hash the bytes they receive. Nobody ever re-serializes to hash. Re-serializing would drag cross-language float formatting — Python repr versus Rust ryu — into the contract; hashing received bytes turns the question into “same file?”, never “same rendering?”. This is the same rule the model hash already follows.

Both backends write the same artifact shape into the same canonical run/data.json — so a fit sampled on jaxstanv5 recomputes to the fingerprint the Rust engine expects.

Match. The fit is accepted; posterior-predictive and posterior-check proceed.
Any byte differs — even whitespace. Refusal is explicit: fit model_data_fingerprint must match the supplied model and data. Repair is to re-run sample, not to patch the artifact.

Where it's pinned

corpus/fingerprints.json

Test vectors: the fingerprint of each reference model's committed document bytes with its canonical data bytes, written by the Python reference implementation.

bayesite fingerprint_vectors

Recomputes every corpus vector in Rust, byte-for-byte, against the same committed files.

artifact corpus

Asserts each committed run directory's posterior.ndjson header and trailer fingerprint equals a recomputation over that directory's own files.

bayescycle end-to-end

Samples on jaxstanv5, then passes bayesite's posterior checks against the real engine — cross-backend, not just cross-language.

trust surface of the default path

Author a model, serialize IR, sample with bayesite: the entire Python environment is bayescycle + bayeswire (stdlib-only), plus one auditable, zero-dependency Rust binary (cargo tree shows a single crate, by CI gate). That binary is the pinned release: bayescycle downloads and sha256-verifies it into ~/.cache/bayescycle/engines/… the first time it runs, then reuses the cache. Every input and output is a file you can hash, diff, and replay.

what CI holds together

Conformance is tested, not promised

The corpus in this repository is the shared baseline: six reference models, their canonical-bytes hashes, JAX-oracle evaluation fixtures, and real run artifacts. Producers must reproduce it byte-for-byte; consumers must evaluate it within the tolerance policy stated in the spec — never in any consumer's tests.

bayeswire every push

  • no-JAX walk — every module imports with jax/blackjax blocked; stdlib-only is enforced, not assumed
  • produce-conformance — reference declarations must reproduce the corpus documents and pinned canonical hashes exactly
  • spec-doc generation — ir-v1-tags.md regenerated from the registry must match the committed file
  • ruff · ty · pytest

jaxstanv5 every push

  • consume-conformance — decode every corpus fixture from the installed pin, bind from the fixture's own data, reproduce logp/grad within the spec's float32 tier (1e-3)
  • statistical oracles (backend-boundary changes) — Stan posterior references and SBC rank-uniformity on real hierarchical models
  • validation loop against the pinned bayeswire

bayesite validation ladder

  • vendored-bytes rung — spec docs and corpus under tests/golden_ir/ must sha256-match bayeswire-vendor.json at the BAYESWIRE_TAG pin; vendored files are generated, never edited
  • fixture evaluation — logp rtol 1e-12, gradient rtol 1e-10 against the float64 JAX-oracle values, plus a bitwise compiled-tape replay check
  • independent NUTS oracle — cross-checked against a pinned nuts-rs; zero-dependency core and wasm build are gates, not goals

bayescycle every push

  • no-JAX job — installs only bayescycle + bayeswire, asserts JAX is absent, downloads the pinned released bayesite engine binary (sha256-verified, via the shared provisioning code — no source build), and runs the end-to-end test: bayeswire model → run directory → sample → diagnose
  • full loop with the [inproc] extra (real jaxstanv5 runs)

bayeswire nightly · on tag

  • cross-repo alignment — checks out each consumer, installs bayeswire HEAD, and runs that consumer's conformance subset: early warning that a pending spec change breaks someone, before a tag is cut. A red nightly blocks tagging, not merging
  • prose audit — greps consumers so normative wire-format prose (and the retired envelope key) can live only here
  • bayesite-head-vs-bayescycle — builds bayesite HEAD from source and runs bayescycle's suite against it; per-PR CI in bayescycle only ever exercises the pinned release, so this nightly job is the only place bayesite HEAD is exercised against a consumer

When the wire format changes — the procedure

  1. Change lands here first. A new distribution or field starts as a corpus reference model and produce-conformance coverage in bayeswire; only then does declaration or backend work begin.
  2. Regenerate the corpus deliberately. scripts/regenerate_corpus.py rewrites documents, hashes, and the generated tag spec. Review every byte of the diff.
  3. Record it. A spec changelog entry plus an explicit IR version decision — bayeswire_ir stays at 1 until a real format change; package versions move freely underneath as long as produce-conformance bytes are stable.
  4. Bump the pins, one PR per consumer. jaxstanv5 and bayescycle move their exact-commit dependency; bayesite re-runs scripts/vendor_bayeswire.py. The pin-bump diff is the compatibility review — a consumer upgrades by reading exactly what changed, with its own conformance gates deciding.
  5. The nightly stands watch. Between pins, cross-repo alignment keeps testing consumers against bayeswire HEAD, so surprises surface as red runs instead of at release time.
Honest asymmetry, worked example. Unifying the corpus revealed that the core profile includes Truncated, which bayesite does not evaluate yet. The gap is not hidden: bayesite pins it as an explicit decode-failure test that names the tag and goes red the moment support lands, evaluation conformance runs on the four supported fixtures, and the work is tracked in bayesite#12. Backends are interchangeable exactly where the conformance suite proves they are — and honestly asymmetric everywhere else.
